Sombre — Complete Read-Only Launch Audit
Read-only. No files modified, no packages installed, no DB writes, no Stripe/email/auth actions. Working tree verified clean at start and end.

1. Executive summary
Sombre is a genuinely well-engineered fragrance ecommerce MVP. The payment/inventory/refund engine, database design, and security posture are production-grade and, in several areas, exemplary — atomic Postgres RPCs with consistent lock ordering, full idempotency against Stripe's at-least-once delivery, RLS with service-role-only private data, 4-layer admin authorization, and clean secret hygiene (zero server-secret leakage into the client bundle). lint, tsc --noEmit, and next build all pass with zero errors and zero warnings. Every prior audit item I could re-verify in code (error boundaries, security headers, shared rate limiting, hero preload, receipt masking, responsive fixes, keyboard a11y) is now implemented and committed.

What blocks real trading is not the code — it is business/legal configuration, deployment, and unproven runtime paths: every legal identity field is an unreplaced [PLACEHOLDER], customers are told to "contact us at [SUPPORT EMAIL ADDRESS]", the app is not deployed, Stripe is in test mode, transactional email is unconfigured (yet the UI promises order-update emails), and the entire payment→webhook→refund→email chain has never executed against real data. There is also no test suite of any kind.

2. Overall launch readiness: ~62%
Engineering foundation ~85%; business/ops/deployment/testing readiness ~30%. The gap is almost entirely non-code.

20. Discount system readiness
Not implemented. Architecture is favorable because pricing is already server-authoritative (route.ts recomputes from DB) — the correct integration seam. Risks to resolve before building it:

Subtotal reconciliation (route.ts:441) compares client vs server subtotal with 0.01 tolerance — a discount must be computed server-side only and folded into this check, never trusted from the client payload.
Stripe line items are built from validated cart items; discounts must map to Stripe discounts/coupon or adjusted unit_amount, and the stored subtotal/total/metadata must record pre/post-discount amounts for later refund math (full vs partial detection in the webhook keys on order.total).
order_items has no discount columns; receipts/emails render unit_price×qty; admin shows totals — all need discount-aware fields.
Provide idempotency, usage limits, code validity, per-product %, and an audit trail (the app currently has no admin action audit log — build that first).
