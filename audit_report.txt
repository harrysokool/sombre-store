Of 11 orders: 10 stock_reduced, 0 stock_restored. The one refunded order is the oversell case, which never reduced stock. E. High priority problems
E1 — A committed source file contains a raw NUL byte (defect I introduced) · verified defect
order-fulfilment-panel.tsx:92 — I used a literal \x00 as a separator instead of the escape sequence. Consequences I confirmed:

file reports the .tsx as data, not text.
Git stores it as binary: the commit shows Bin 0 -> 10482 bytes, so there is no line diff, no blame, no merge for a 340-line file.
grep -r silently skips it. Searching src/ for isFulfilmentTransitionAllowed returns only the definition; the three call sites are invisible without grep -a.
It compiles and runs correctly, which is why lint, tsc, and build all pass. Fix: replace the byte with \u0000 (identical runtime value, ASCII source).

E2 — Success page polls forever · partially implemented
checkout-success-state-manager.tsx:35-39 refreshes every 3s while an order is pending or refund_pending, with no attempt cap and no backoff. Each refresh triggers a live stripe.checkout.sessions.retrieve plus two Supabase queries. A stuck order with an open tab = 1,200 Stripe API calls/hour, indefinitely.

E3 — Rate limiting is ineffective on the likely deployment target · partially implemented
rate-limit.ts:1-6 is per-process in-memory, and honestly documents this. On Vercel/serverless each invocation may get a fresh map, so the 10/min checkout limit is close to unenforced. Checkout creates Stripe sessions, so this is an abuse and cost vector.

E4 — No security headers at all · not implemented
next.config.ts:3 is an empty object. No HSTS, CSP, X-Frame-Options, X-Content-Type-Options, or Referrer-Policy. Note the referrer gap compounds E5: the success URL carries the session ID.

E5 — Receipt exposes full customer PII to anyone holding the session ID · partially implemented
success/page.tsx:165-178 renders name, email, phone, and full address. Verification itself is strong (receipt.ts:152-168 cross-checks payment intent, currency, and all three amounts), but there is no expiry and no Referrer-Policy, so a shared or leaked URL discloses a home address permanently.

E6 — HK district flow has never completed end-to-end · implemented but untested
district is NULL on all 11 orders — every one predates the column. Form (required), server validation (route.ts:55), Stripe metadata, and webhook write all look correct, but the chain has never actually run. Postal code is correctly optional on both sides.

E7 — No error boundaries · not implemented
No error.tsx, global-error.tsx, or not-found.tsx anywhere in src/app. Any server error shows Next's raw default page mid-checkout.
